## Architecture

### Routing

TanStack Start file-based routing. Routes live in `src/routes/`. Only `__root.tsx` (app shell) and `index.tsx` (Gantt page) exist. Do NOT create `src/pages/`, `_app/`, or `app/layout.tsx` — those are Next.js/Remix conventions.

- **`routeTree.gen.ts`** is auto-generated by TanStack Router. Never edit it by hand.
- **Server entry**: `src/server.ts` wraps the SSR handler with h3 error recovery.
- **Start instance**: `src/start.ts` creates the TanStack Start app with error middleware.

### State management & persistence

All state is client-side using a manual `useSyncExternalStore` + localStorage pattern. There is no database or external API.

Two stores in `src/lib/`:

| Store    | File                | localStorage key    | Interface         |
| -------- | ------------------- | ------------------- | ----------------- |
| Tasks    | `gantt-store.ts`    | `gantt-tasks-v1`    | `Task`            |
| Settings | `settings-store.ts` | `gantt-settings-v1` | `ProjectSettings` |

Both stores follow the same lifecycle:

1. **Hydration**: On first client-side access, loads from localStorage synchronously (fast path). Seeds default data if localStorage is empty.
2. **Server sync (load)**: After hydration, fetches from the server via `getProjectData()`. If server has data, overwrites local state and notifies listeners. If server has no file yet, saves current state to server.
3. **Server sync (save)**: Every mutation (`add`, `update`, `remove`, `addComment`, `settingsStore.update`) calls `persist()` which writes to localStorage and fires `mergeProjectData()` asynchronously.

### Server functions (JSON persistence)

**This version of TanStack Start (v1.168.x) does NOT export `createAPIFileRoute`** (no `@tanstack/react-start/api` subpath). Use `createServerFn` from `@tanstack/react-start` instead.

Server-side persistence lives in `src/lib/json-persist.ts`:

- **`getProjectData`** — `createServerFn({ method: "GET", strict: false })` that reads `project-data.json` from `process.cwd()`. Returns `null` if file doesn't exist.
- **`mergeProjectData`** — `createServerFn({ method: "POST", strict: false })` with a `.validator()` that accepts `Partial<ProjectData>` and merges with existing file data (only overwrites the field sent; preserves the other).

The JSON file (`project-data.json` at project root) has this structure:

```json
{
  "tasks": [Task],
  "settings": { "name": "...", "startDate": "YYYY-MM-DD", "endDate": "YYYY-MM-DD" }
}
```

**Important**: The `strict: false` option is required on both functions because TanStack Start's serializability checker rejects `unknown[]` and `Record<string, unknown>` types otherwise.

**Race condition note**: Both stores send only their own slice (`{ tasks }` or `{ settings }`) to `mergeProjectData`. The server merges with existing file data, so concurrent writes from both stores don't clobber each other. Do NOT make a GET before POST in the stores — that creates a race condition.

### Gantt chart

`src/components/gantt/GanttChart.tsx` — the main timeline component.

- **Weeks start Monday, end Friday**: The timeline snaps the start date back to the previous Monday. Weekends (Sat/Sun) are excluded from the timeline.
- **Sprints**: Two calendar weeks (Mon-Fri × 2 = 10 workdays) grouped as one sprint. Sprint headers show date ranges.
- **Day grid**: Each weekday is a 40px column. Friday columns have a stronger right border.
- **Today line**: A vertical red line marks the current day if it's within the visible range.
- **Bar colors**: `status-progress` (blue), `status-complete` (green), `status-blocked` (red), `status-partial` (orange). Defined as CSS variables in `src/styles.css`.

The chart accepts `projectStart` and `projectEnd` props from settings, which define the visible timeline range. When the user updates project dates in the Settings dialog, the chart re-renders with the new range.

### Task model

```ts
interface Task {
  id: string;
  parentId: string | null; // null = top-level
  title: string;
  assignee: string;
  startDate: string; // YYYY-MM-DD (planned)
  endDate: string; // planned
  actualStartDate?: string; // real
  actualEndDate?: string; // real
  progress: number; // 0..100
  block: "none" | "partial" | "total";
  blockReason?: string;
  comments: Comment[];
  createdAt: string;
}
```

All fields are persisted to `project-data.json` on every change.
